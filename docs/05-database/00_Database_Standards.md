# Database Standards

**Project:** ClinicOS – AI Workflow Platform for Healthcare Clinics

**Version:** 1.0

**Status:** Approved

**Owner:** Database Architecture Team

---

# 1. Purpose

This document defines the database standards for ClinicOS.

Every table, relationship, index, migration, and query must follow these standards.

The objective is to ensure consistency, maintainability, scalability, and security across the entire platform.

---

# 2. Database Technology

Primary Database

* PostgreSQL

ORM

* Prisma ORM

Migration Tool

* Prisma Migrate

Local Development

* PostgreSQL (Docker / Neon)

---

# 3. Naming Convention

## Tables

Use **snake_case** and plural names.

Examples

```text
users
patients
appointments
doctors
clinics
roles
permissions
notifications
```

---

## Columns

Use snake_case.

Example

```text
first_name
last_name
phone_number
created_at
updated_at
tenant_id
doctor_id
patient_id
```

---

## Prisma Models

Use PascalCase.

Example

```text
User

Patient

Doctor

Appointment
```

---

# 4. Primary Keys

Every table uses UUID.

```sql
id UUID PRIMARY KEY
```

Never use auto-increment integers.

---

# 5. Foreign Keys

Foreign keys follow:

```text
tenant_id

clinic_id

doctor_id

patient_id

appointment_id

user_id
```

Never abbreviate IDs.

---

# 6. Audit Fields

Every transactional table contains:

```text
id

created_at

updated_at

created_by

updated_by
```

Optional

```text
deleted_at

deleted_by
```

Soft delete is preferred over hard delete.

---

# 7. Multi-Tenant Rule

Every business table contains

```text
tenant_id
```

Examples

* doctors
* patients
* appointments
* conversations
* insurance
* documents

System tables may omit tenant_id.

---

# 8. Status Fields

Avoid multiple boolean flags.

Instead use enums.

Bad

```text
is_active

is_deleted

is_verified
```

Good

```text
status

ACTIVE

INACTIVE

SUSPENDED
```

---

# 9. UUID Standard

UUIDs are generated by the backend.

Never generated by the frontend.

---

# 10. Time Standard

Store all timestamps in UTC.

Frontend converts to local timezone.

---

# 11. Relationships

Use foreign keys.

Never duplicate information.

Bad

```text
doctor_name
```

Good

```text
doctor_id
```

---

# 12. Soft Delete

Critical business tables use soft delete.

Examples

* patients
* doctors
* appointments
* clinics

Reference tables

Example

roles

can use hard delete if appropriate.

---

# 13. Constraints

Always define

* NOT NULL
* UNIQUE
* CHECK
* FOREIGN KEY

Examples

Email

Unique

Doctor Availability

No overlapping slots

Appointment Time

Cannot be in the past (business rule)

---

# 14. Indexing Standards

Every table

Primary Key

Unique indexes where required

Indexes on

```text
tenant_id

status

created_at

updated_at
```

Business indexes

```text
doctor_id

patient_id

appointment_date

clinic_id
```

Composite indexes when required.

---

# 15. Enum Strategy

Use enums for controlled values.

Examples

AppointmentStatus

```text
SCHEDULED

CONFIRMED

COMPLETED

CANCELLED

NO_SHOW
```

Gender

```text
MALE

FEMALE

OTHER

PREFER_NOT_TO_SAY
```

---

# 16. JSON Usage

Use JSONB only for flexible data.

Examples

```text
notification_metadata

ai_metadata

workflow_context
```

Business entities should remain relational.

---

# 17. Large Files

Never store files inside PostgreSQL.

Store only

```text
file_name

file_url

mime_type

size
```

Actual file

↓

MinIO / AWS S3

---

# 18. AI Data

PostgreSQL stores

* metadata
* prompts
* request history

Vector embeddings remain in ChromaDB.

---

# 19. Migration Policy

All schema changes

↓

Prisma Migration

↓

Git

↓

CI/CD

↓

Deployment

No manual production schema changes.

---

# 20. Backup Policy

Development

Manual

Production

Automated

Point-in-Time Recovery

Encrypted Backups

---

# 21. Security

No secrets inside database.

Passwords

↓

bcrypt

Sensitive values encrypted where required.

Always use parameterized queries.

---

# 22. Coding Standards

Repositories are the only layer allowed to communicate with Prisma.

Controllers never access Prisma.

Services never contain SQL.

---

# 23. Future Compatibility

The schema must support:

* Multi-region deployment
* Read replicas
* Event sourcing
* CQRS
* Microservice extraction
* Data warehouse integration

without redesign.

---

# Conclusion

These standards are mandatory for every entity created in ClinicOS.

Following them ensures consistency, maintainability, and scalability as the platform grows from an MVP into an enterprise healthcare SaaS platform.
